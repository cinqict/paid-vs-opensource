Paid vs Open Source Comparison Project

By James Twose

Frameworks to be compared

Databricks Workspaces

  • Monolithic architecture
  • Linear like (with feedback) flow from data to final result
  • Technologies used: Azure databricks, Databricks Delta Lake, Apache Hive, Python, Pyspark, Pandas, xgboost, mlflow, streamlit

Kubernetes

  • Distributed/ Microservice architecture
  • Technologies used: Kubernetes, Minikube, docker, MySQL, Python, Pandas, scikit-learn, xgboost, FastAPI, streamlit (Optional: Azure Kubernetes Service (AKS))

Project

What is the project we are using for this comparison?

  • Using weather (temperature and rain amount) to predict amount of time train is delayed
  • Train on historical train disruption data (rijdendetreinen.nl)
  • Use historical processed weather data (open-meteo.com) as features
  • Host the trained model so it can be queried
  • Create a dashboard showing the weather and comparing the prediction vs what NS reports

Azure databricks (paid)

Kubernetes (open source)

Discussion

Databricks Workspaces

  • Quite rigid in terms of what you can do
  • Quick learning curve, comprehensive documentation
  • A lot of add ons (model registry, experiments, deployment, etc.)
  • Minimal difference between azure databricks and azure machine learning

Kubernetes

  • Very flexible
  • Somewhat confusing initially, documentation sometimes out of date
  • Can be used for a lot of different things (also due to a lot of parts of the databricks workspace being open source, possible to emulate)
  • Less expensive to host
